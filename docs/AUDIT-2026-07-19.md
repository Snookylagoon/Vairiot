# Vairiot Platform Audit — 19 July 2026

Full-stack audit covering **Security**, **SaaS capability**, **Online/Offline ability**, **Communications**, **Storage**, and **Infrastructure/Hosting**. Read-only — no code was modified. Evidence is cited as `file:line`.

---

## Executive summary

**The architecture is genuinely SaaS-grade; the risk is concentrated in operations and offline edge cases.**

- **Strong foundation**: first-class multi-tenancy (tenant-scoped schema, service-layer isolation verified across sampled routes), licence tiers with enforced asset caps, self-serve registration + onboarding wizard, sub-tenant hierarchy, API keys + OpenAPI docs, BullMQ job queue, S3-compatible object storage, bcrypt-12 + 2FA + progressive lockout + JWT blacklist, TLS 1.2/1.3 only, no secrets committed to git.
- **Existential gap**: there are **no backups of any kind**. Postgres, MinIO (all photos/documents/APKs), Redis, TLS certs, and the production `.env` (including `APP_ENCRYPTION_KEY` and `JWT_SECRET`) live on a single VPS disk. If that disk dies, the company's data is gone permanently (RPO = infinity).
- **Active data loss in the field**: offline sync silently deletes queued work after 5 failed attempts, and Android blind-audit scans queued offline are **always** rejected on replay (missing `locationId`) — a crew auditing in a dead zone today loses every scan, permanently and silently. Offline photo capture discards the photo.
- **Nobody is watching production**: no uptime monitoring, no error tracking, no alerting, no log rotation. The first alert for a 2am outage is a customer.
- **Hosting location is right, hosting shape is wrong**: the production VPS is in Europe, which suits the Europe/UAE client base — but all state sits on one unmanaged disk. The move is to managed EU services, not a new region; UAE-regulated tenants may later need an in-country deployment option.

## 1. Security

**Posture: above average for project size. 0 critical in production paths, 2 high, 6 medium, 5 low.**

### High
| ID | Finding | Evidence | Fix |
|----|---------|----------|-----|
| H1 | `nodemailer <= 9.0.0` — SMTP command injection / SSRF / file-read advisories; the only production-relevant npm audit hit | `vairiot-api/src/services/smtp.service.ts:119` | Upgrade to `nodemailer@^9.0.3`, smoke-test SMTP verify/test-email flow |
| H2 | iOS UDID enrolment endpoint is public and parses the PKCS#7 plist **without verifying the signature** — forged device attributes can populate the enrolment queue | `vairiot-api/src/routes/ios/ios.router.ts:211-235`, mounted publicly at `app.ts:68` | Verify PKCS#7 signature against Apple's CA, rate-limit, treat queue as untrusted |

### Medium
| ID | Finding | Evidence | Fix |
|----|---------|----------|-----|
| M1 | 2FA TOTP secrets and backup codes stored **plaintext** in DB (AES-256-GCM helper exists but is only used for SMTP creds) | `two-factor.service.ts:25-35` vs `lib/crypto.ts` | Encrypt secrets with existing `encryptSecret()`; hash backup codes |
| M2 | Password policy requires **exactly** 12 chars, alphanumeric **only** (symbols rejected); login validates min 8, registration min 12 — inconsistent | `password-policy.service.ts:3-20`, `auth.router.ts:22,52` | Minimum (not exact) length, allow all printable chars, add breach-list check |
| M3 | API/worker run with **MinIO root credentials** — a compromised API owns the whole object store | `lib/minio.ts:12-13`, `docker-compose.prod.yml:70-71` | Scoped MinIO service account limited to the three app buckets |
| M4 | Refresh-token rotation never invalidates the presented token — stolen refresh token stays valid 30 days in parallel | `auth.service.ts:136-147`, `lib/jwt.ts:11` | Blacklist presented `jti` on refresh; add reuse detection |
| M5 | Single shared JWT base secret for access/refresh/setup token classes; placeholder secrets in local envs | `lib/jwt.ts:6-9` | Distinct high-entropy secrets per token class; assert min length at startup |
| M6 | `APP_ENCRYPTION_KEY` missing from example env files; static scrypt salt; ≥16-char minimum too low | `crypto.ts:5,11-14` | Document in `.env.example`, require ≥32 chars |

### Low
- L1: APK/IPA download endpoints fully public (accepted trade-off for pre-login OTA; add rate limiting).
- L2: **No HSTS** on the nginx SPA vhosts (Helmet covers API only) — one-line `add_header` in `infra/nginx/prod.conf` 443 blocks.
- L3: Android tokens in plain DataStore (`TokenStore.kt:15`); iOS correctly uses Keychain — migrate to EncryptedSharedPreferences/Keystore.
- L4: No certificate pinning in either mobile app (TLS enforced, cleartext disabled; pin the CA key if hardening).
- L5: `LabelsPage.tsx:488` uses `innerHTML` for print content — confirm asset-derived strings are escaped.

### Done well
No secrets ever committed (verified via git history); bcrypt cost 12 with dummy-compare against user enumeration; progressive lockout + per-route login limiter; JWT hard-fails if unset; logout blacklists both tokens via Redis; coherent RBAC + platform-admin separation; tenant isolation threaded through every service query; no SQL injection surface (Prisma everywhere, one static raw query); CORS allowlist; API keys stored as SHA-256 hashes; Android OTA verifies SHA-256 of downloaded APKs; private buckets, all files streamed through authenticated API.

## 2. SaaS capability

**Verdict: designed for SaaS from day one (deploymentMode enum, licence tiers, sub-tenants, onboarding wizard). Gaps are operational, not architectural.**

### What exists
- **Multi-tenancy**: `Tenant` model with plan/featureFlags/parent-child hierarchy; every business table carries `tenantId` with composite indexes; services consistently use `findFirst({ id, tenantId })` — cross-tenant ID probing returns 404 in all sampled routes. MinIO keys are tenant-prefixed. Sub-tenant context switching mints scoped JWTs.
- **Provisioning**: self-serve `POST /auth/register` (tenant + roles + admin in one transaction), 4-step onboarding gate, invitation tokens with emailed invites, platform-admin plane with transactional tenant deletion (~30 tables + MinIO prefix purge).
- **Licensing**: FREE/TIER_2/TIER_3 with `maxAssets`, device slots, pricing, expiry/grace lifecycle; `enforceAssetCap()` blocks creation over quota.
- **Public API**: `/api/v1` versioning, OpenAPI at `/api/docs`, tenant-scoped hashed API keys, per-tenant webhooks.

### Gaps (ranked)
| # | Gap | Effort |
|---|-----|--------|
| 1 | **No payment gateway** — `paymentConfirmed` is flipped manually by a platform admin; no Stripe/Paddle anywhere. Licence state machine already built → Stripe Billing is integration, not redesign | M |
| 2 | **No staging environment**; schema changes effectively tested on prod | M |
| 3 | **Migrations not wired into deploy** — `db:deploy` script exists but nothing calls it | S |
| 4 | **In-memory rate limiting & lockout counters** — blocks running >1 API replica; swap to `rate-limit-redis` (Redis already deployed) | S |
| 5 | **No DB-level tenant isolation** — one forgotten `where: { tenantId }` in future code = cross-tenant breach. Add a Prisma client extension auto-injecting tenant scope + CI cross-tenant fuzz test (or Postgres RLS) | M/L |
| 6 | Webhook delivery is fire-and-forget fetch — no retry, no HMAC signing in dispatch, no delivery log. Route through BullMQ | S/M |
| 7 | Mobile server URL compile-time hardcoded (`build.gradle.kts:42`, `APIClient.swift:48`) — blocks hybrid/self-hosted deals the `deploymentMode` enum implies | S/M |
| 8 | No usage metering (storage per tenant, API calls) — aggregate `sizeBytes` nightly, enforce tier storage caps | S |
| 9 | White-label thin (logo + currency only); SMTP is a platform-global singleton | M |
| 10 | OTA release channel is global — a bad build hits 100% of tenants; add rollout % or per-tenant pinning | S |

## 3. Online/Offline ability

**Verdict: solid v1 offline architecture on both platforms, but several silent data-loss paths that must be fixed before field crews rely on it.**

### Architecture
- **Android**: Room DB (queued scans/assets, cached assets, RFID scan sessions); WorkManager periodic 15-min sync with CONNECTED constraint + on-failure one-shots (survives reboot). Audit scans use write-ahead queueing (crash-safe).
- **iOS**: SwiftData; `NWPathMonitor` drains queue on reconnect/foreground; provisional cache rows make queued creates visible immediately; network errors stop the drain without burning attempts (smarter than Android); keeps session when token refresh fails offline.
- **Server**: only scan-session upload is sync-aware (idempotent upsert on client UUID). Everything else is plain REST replay.

### Data-loss findings (ranked)
| # | Finding | Impact | Fix | Effort |
|---|---------|--------|-----|--------|
| 1 | **Drop-after-5-attempts silently deletes queued work** on both platforms; Android's 30s backoff can exhaust all 5 in minutes; Android's cold-start token wipe (`VairiotApp.kt:34`) means post-reboot workers 401 and burn attempts before the user can log in | Permanent silent loss of scans/creates | Never delete on failure — add a `dead` state + "failed items" retry/discard UI; don't count network errors as attempts; skip sync when unauthenticated | M |
| 2 | **Android blind-audit offline scans are always rejected**: `AuditRunViewModel.kt:157-159` queues without `locationId`/`condition` (columns exist, unpopulated); server rejects (`audit.service.ts:169-171`) → 5 fails → dropped. Conditions silently discarded on both platforms | 100% loss of offline blind-audit data | Persist + replay both fields — near two-line fix each side | **S** |
| 3 | **No idempotency on replay**: crash between API success and queue-delete duplicates assets/scan events; no unique constraint on barcode/rfidTag; `nextAssetNumber` read-then-increment race | Duplicate records, inflated audit counts | Send client UUID as `clientRequestId` (iOS already generates one); server dedupe + return-existing-on-conflict | M |
| 4 | **Offline photo capture loses the photo** — no queue, no gating; upload fails, photo discarded | Field data loss | Queue photo files on disk with a `QueuedPhoto` row, drain in existing workers (or minimally gate capture offline) | M / S |
| 5 | Android session lifecycle defeats offline: cold-start token wipe + `tokenStore.clear()` on any refresh 5xx (`NetworkModule.kt:56-58`) | Offline cold start lands on unusable login screen | Port iOS policy: clear only on genuine rejection | S |
| 6 | No iOS background sync (foreground-only) | Queued data stranded until next app open | Add `BGProcessingTask` | M |
| 7 | Replayed scans get server-stamped timestamps (no `capturedAt` field) — audit-evidence integrity | Wrong audit timestamps | Add `capturedAt` with server-side sanity clamp | S |
| 8 | Full-register re-download on every refresh; no delta sync; cache age tracked but never shown | Bandwidth + stale decisions | `GET /assets?changedSince=` (uses existing `updatedAt`); "last synced X ago" label | M / S |
| 9 | Platform parity: Android lacks reference-data cache (offline creates can't set site/category — iOS-only since b830bbf), provisional rows, connectivity monitor; iOS lacks RFID sessions, blind-audit zones, pending-count UI; iOS provisional-row orphan/vanish edge cases | Feature gaps | Port each direction | S–M |

Web/admin are fully online-only (no service worker) — acceptable, note for roadmap.

## 4. Communications

- **API**: Express with helmet, CORS allowlist, request-IDs, consistent `AppError` hierarchy, Swagger UI, versioned routes. Validation via express-validator is per-route and uneven (not schema-first). No compression anywhere (mobile pulls uncompressed JSON over cellular). No server/proxy timeouts tuned. Global limiter 100 req/min per IP will throttle NAT'd scanner fleets flushing offline queues.
- **Email**: nodemailer/Resend via worker (invites, audit-complete + CSV, alert digests). SMTP config encrypted in DB.
- **Broken silent feature — alert digests never fire**: subscription CRUD exists, the worker consumer is fully built, but **no producer/scheduler ever enqueues the job** (`lib/queue.ts` defines only audit-complete and user-invite). Users can subscribe to alerts that will never arrive. Maintenance-due detection is on-demand only — no proactive notifications at all.
- **Push/SMS**: none. Mobile relies on unauthenticated version polling for OTA.
- **Real-time**: none; nginx `/ws/` upgrade blocks are dead config.
- **Worker**: BullMQ, 3 attempts + exponential backoff — but no dead-letter alerting (failed jobs silently evicted after cap 200) and enqueue failures are swallowed with a log line.

## 5. Storage

- **Postgres**: 45 models, well-indexed hot paths (barcode/rfid/tenant composites), 32 proper Prisma migrations — but nothing runs `migrate deploy` on deploy. Default Prisma pooling (fine at current scale). **No backups.**
- **MinIO**: three buckets (photos, documents, mobile-releases), tenant-prefixed keys, sensible upload caps (10 MB photos + MIME allowlist, 25 MB docs, 150 MB APKs). Last-3 release retention implemented app-side. No presigned URLs — every photo/APK streams through the API (memory/CPU + rate-limit budget). No lifecycle rules, no versioning, **no backups**. Delete failures swallowed → orphaned blobs; no orphan sweep; asset hard-delete paths don't clean MinIO.
- **Volumes**: everything on bind mounts under `/opt/Vairiot/infra/data/*` on one host. Host disk death = **total loss**: all tenant data, all files, all releases, plus the un-versioned `.env` — losing `APP_ENCRYPTION_KEY` also orphans encrypted SMTP creds even if a dump existed.
- **Lifecycle**: soft deletes only on User/Asset; solid `AuditEvent` log (unbounded growth, no retention); CSV/report exports exist but **no whole-tenant export** (SaaS offboarding/GDPR); `ReportSchedule` model is a dead stub; no server-side image processing (thumbnails client-generated, no EXIF/GPS strip).

## 6. Infrastructure & operations

- **Deploy**: fully manual `ssh → deploy.sh` doing `git pull` + `compose up -d --build` — images **built on the prod box**, no registry, no tags, **no rollback**, not zero-downtime. nginx caches upstream container IPs and must be manually restarted after every deploy (documented 502 foot-gun) — `resolver 127.0.0.11 valid=10s;` + variable `proxy_pass` removes the whole failure class.
- **CI**: typecheck + API tests + Android lint/build only. No ESLint run, zero coverage for worker/admin/shared/reports, no Docker build check, CI on Node 24 vs prod Node 22, and CI doesn't gate deploys.
- **Resilience**: `restart: always` everywhere (good); api/worker/web/admin/nginx have **no healthchecks**; **no resource limits** (one leak OOMs the box including Postgres); **no log rotation** (json-file grows until disk-full).
- **Observability**: console-only logging, in-memory metrics reset on restart, `/health` + `/health/ready` exist but nothing consumes them. **No Sentry, no uptime monitor, no alerting.**
- **TLS**: solid nginx config (1.2/1.3, ACME webroot, 444 for unknown SNI) — but no certbot deploy-hook to reload nginx, so a renewed cert isn't picked up until the next manual restart: a latent expiry incident on any quiet ~60-90 day stretch.
- **Pinning**: `minio/minio:latest` unpinned in all three compose files (breaking-change landmine); Prisma 5.22 Node-22 issue forces the CI/prod Node mismatch; no Dependabot/Renovate.

## 7. Hosting assessment & SaaS access

**Client base: Europe and UAE.** The current VPS (81.85.92.155) geolocates to UK/EU ranges, so it is **already well-placed geographically**: European clients get low RTT, and UAE clients see ~120–160 ms to Europe — workable for the API, further softened by the offline-first mobile design. Relocation for latency is *not* required. The drivers for hosting change are instead:

- **Resilience** — all state on one unmanaged disk (see §5/§6). Moving Postgres and object storage to managed EU services is the priority, not moving regions.
- **GDPR** — EU hosting is the right default for European clients; ensure a DPA with the hosting provider and keep backups inside the EU/UK.
- **UAE PDPL (Federal Decree-Law 45/2021)** — permits cross-border transfer with adequate protection/contractual safeguards; private-sector UAE clients are generally fine on EU hosting. However, UAE government/semi-government and regulated sectors (banking under Central Bank rules, health data under ICP in-country requirements) may require **in-country hosting**. The platform's existing `deploymentMode` (standalone/saas/hybrid) is the right vehicle: offer a UAE-region deployment (AWS me-central-1 Dubai, Azure UAE North, or local providers such as Khazna/G42) as a per-tenant option when procurement demands it.
- **Front-end access** — put a CDN (e.g. Cloudflare) in front of the SPAs and photo delivery to improve UAE load times cheaply.

**Options (recommended order):**

| Option | Shape | Cost (USD/mo) | Migration effort |
|--------|-------|---------------|------------------|
| **A (recommended now)** | Stay in an EU region; keep (or upgrade) the app VPS + **managed Postgres** + S3-compatible object storage replacing self-hosted MinIO (API already speaks S3 — config change). e.g. Hetzner/current VPS + AWS eu-central-1 RDS + S3, or Scaleway/OVH equivalents. Compose largely unchanged | ~60–150 | **M** — pg_dump/restore + `mc mirror` + DNS cutover; one evening of downtime. Kills the single-disk blast radius without a region move |
| B | Managed platform — Fly.io (Frankfurt/London/Amsterdam) or Render (Frankfurt) with CI-built images, health-checked rollouts, instant rollback; Neon/managed Postgres + Tigris/S3 | ~80–250 | M–L — platform routing replaces nginx/compose |
| C | Full cloud — AWS eu-central-1 (or eu-west-2) primary: ECS/EC2 + RDS (PITR, multi-AZ) + S3 + ALB + CloudFront; add a **me-central-1 (Dubai)** satellite deployment later for UAE-residency tenants | ~250–600+ (per region) | L — worth it when tenant count or UAE/government procurement demands it |

Do the CI-built-images work first so whichever host you land on receives images, not source builds.

## 8. Prioritised remediation roadmap

### This week (mostly S effort — existential/data-loss)
1. **Off-host backups**: nightly `pg_dump -Fc` + `mc mirror` of MinIO + `.env` copy to another provider/region (B2 or S3 ap-southeast-2, ~$1–5/mo); store `APP_ENCRYPTION_KEY`/`JWT_SECRET` in a password manager; document + test restore.
2. **Fix Android blind-audit offline scans** (persist/replay `locationId` + `condition`) — near two-line fix, currently 100% data loss.
3. **Uptime + error monitoring**: external check on `/health/ready` (UptimeRobot/Better Stack) with phone alert; Sentry free tier across api/worker/web/admin.
4. **Upgrade nodemailer** (H1).
5. Compose hygiene: pin `minio/minio`, add `logging` max-size/max-file, add `mem_limit` per service; nginx `resolver` fix + certbot `--deploy-hook` reload; add HSTS to SPA vhosts; gzip on.

### This month (M effort — stability & trust)
6. Wire `prisma migrate deploy` into `deploy.sh`/entrypoint; add healthchecks for api/worker/nginx.
7. Offline sync integrity: dead-letter state instead of delete-after-5; don't count network errors; skip sync unauthenticated; port iOS token-retention policy to Android; add `clientRequestId` idempotency for creates/scans.
8. Security mediums: encrypt 2FA secrets + hash backup codes; refresh-token rotation invalidation; scoped MinIO service account; fix password policy; separate JWT secrets.
9. Build the missing scheduler: BullMQ repeatable job for alert digests + maintenance-due notifications (UI already promises them).
10. CI hardening: ESLint, worker/admin/shared/reports coverage, Docker build checks, align Node versions (bump Prisma 6.x), Dependabot; build images in Actions → GHCR → `compose pull` deploys with instant rollback.

### This quarter (L effort — SaaS growth)
11. **Move state to managed EU services** (Option A: managed Postgres + S3-compatible storage in an EU region); add a CDN in front of the SPAs; scope a UAE-region deployment option for regulated tenants.
12. Stripe Billing onto the existing licence state machine; nightly per-tenant storage metering.
13. Staging environment; Prisma tenant-guard extension + cross-tenant CI fuzz test.
14. Offline photo queue; delta sync (`changedSince`); iOS background sync; platform parity items.
15. Whole-tenant data export; webhook delivery via BullMQ with HMAC signing + delivery log; OTA staged rollout; per-tenant SMTP/branding.
